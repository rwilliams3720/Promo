DIRECTIVE 2 — MANUAL SALES ENTRY & SALES TRACKING ADD-ON
================================================================================

WHAT THE DIRECTIVE ASKS FOR:
- Research the three sample XLSX files (buich jan.xlsx, Justin sample.xlsx,
  admin all product sales April 2026) to understand data shapes
- Build a UI to manually enter sales data and store it in Supabase (same sales_log table)
- Manual entries must be editable after the fact
- Privacy rule: only store first 2 initials of customer's first name + full last name
  (e.g., "Jo Smith" not "John Smith")
- Toggle to switch between upload mode and manual entry mode
  - Toggle OFF upload: deletes currently uploaded data, replaces with manual
  - Toggle ON upload: resumes file-based flow and deletes manual uploaded
  - Both directions require a confirmation warning, ideally recommending action
    at the start of a new month or to save and archive before proceeding
- Sales tracking (manual entry mode) is a $25/month add-on available at any tier
- Account tab needs an interface to add the sales tracking add-on to an existing plan
- Need directions for Stripe subscriptions that can't be auto-handled. Secondary subscription recommended that can be turned off if desired and sales tracker access tied to status.

GAPS & LOGIC ISSUES TO RESOLVE BEFORE BUILDING:

1. SOURCE COLUMN — sales_log has no "source" field distinguishing upload vs. manual.
   If toggle switching deletes one set and inserts another, this works by full
   replacement. But if we ever want to preserve both or audit which rows came from
   where, we'll need to add a `source` column (values: 'upload' | 'manual') to
   sales_log. Decision needed: delete-and-replace (simple) vs. source-tagged rows
   (auditable but more complex). Recommend delete-and-replace per month, matching
   the existing upload behavior, unless the user needs mixed sourcing.

2. MANUAL ENTRY UI — The three sample files need to be opened and column structures
   mapped before building the form. Known product columns from the directive context:
   WL, UL, Term, Health, Auto, Fire. The sales_log insert schema needs to match
   whatever upload.js writes. Open all three files and list every column header
   before designing the form fields.

3. CUSTOMER NAME PRIVACY RULE — First 2 initials of first name + full last name.
   Implementation: on form submit, strip the first name down to initials before
   storing. Example transform: "Jonathan" → "Jo". This should happen client-side
   before the API call so no full name ever transits the wire. No server-side PII
   handling needed, but the form label must communicate this clearly to the user.

4. AI ANALYSIS SOURCE PRESERVATION — The directive says: confirm that no matter
   how data is uploaded or entered, the analyzer fetches from the active source
   while preserving historical fetches from previously used sources.

   Current behavior: ai-analysis.js reads from sales_log and historical_months.
   historical_months is written at archive time from whatever was in sales_log then.
   So if a month was tracked by upload, archived, then the user switches to manual
   for the next month, the archived month retains its upload-sourced data — that's
   already correct.

   What's NOT safe: switching modes mid-month (toggle off upload, replace with
   manual entries) should not corrupt an already-archived month. Since archive
   copies sales_log → historical_months and then clears sales_log, the switch
   only affects the current month. This is fine as long as the confirmation warning
   is shown before toggle.

   No code change to ai-analysis.js should be needed, but add a note in the
   toggle confirmation: "Switching modes will replace this month's sales data.
   Archived months are not affected."

5. TOGGLE IMPLEMENTATION DETAIL — Where does the toggle state live?
   Options:
   a) In-memory only — reset on page load (bad, user would have to re-toggle)
   b) In accounts table — add a column `sales_entry_mode` ('upload' | 'manual')
   c) In race_config — key/value pair for the current user

   Recommend: accounts table column. It's per-user persistent state that the
   backend and frontend both need to read. race_config is for race-level data,
   not user preferences. SQL: ALTER TABLE accounts ADD COLUMN sales_entry_mode
   text NOT NULL DEFAULT 'upload';

   The upload endpoint (api/upload.js) should check this flag when receiving a
   sales upload — if mode is 'manual', reject with an error message rather than
   silently overwriting.

6. ADD-ON BILLING — $25/month for sales tracking.

   What needs to be built:
   a) Supabase: accounts table needs a `has_sales_addon` boolean column
      (DEFAULT false). This is the gating flag the frontend reads.
   b) Stripe: a new Price object for the $25 add-on (one-time setup via Stripe
      dashboard). The price_id needs to go in a Vercel env var.
   c) api/stripe-checkout.js or a new api/addon-checkout.js: creates a Stripe
      Checkout session for the add-on, attaches it to the existing customer.
   d) api/stripe-webhook.js: on subscription activation for the add-on price,
      set accounts.has_sales_addon = true. On cancellation, set false.
   e) Account tab UI: show a panel if !has_sales_addon with a button to add it.
      If has_sales_addon, show current status and a cancel option.

   MANUAL STRIPE STEPS (for subscriptions that can't be auto-handled):
   - Log in to Stripe Dashboard → Customers → find by stripe_customer_id
   - Click the customer → Subscriptions → Add product
   - Select the Sales Tracking add-on price ($25/month)
   - Save. The webhook will fire invoice.payment_succeeded → set has_sales_addon=true
   - If the customer already paid out-of-band: manually set has_sales_addon=true
     via Supabase SQL: UPDATE accounts SET has_sales_addon=true WHERE email='...';

   NOTE: The stripe_customer_id column already exists on the accounts table.
   The webhook handler (api/stripe-webhook.js) already exists and handles
   invoice events. Extending it for the add-on is the right approach vs. new file.

7. CRUD API FOR MANUAL ENTRIES — Manual sales will need:
   - GET /api/sales — fetch current month's manual entries for the user
   - POST /api/sales — insert a new manual entry (validates field types, applies
     privacy transform to customer name)
   - PATCH /api/sales?id= — update an existing entry
   - DELETE /api/sales?id= — delete an entry

   Alternative: since sales_log already has a hash PK and insert behavior, a single
   /api/sales endpoint with GET/POST/PATCH/DELETE methods is cleaner than four routes.
   The hash for manual entries should be a SHA-256 of the agent+date+products to
   prevent accidental duplicates, same approach as call dedup.

8. FEATURE GATING — Manual entry toggle should only appear if has_sales_addon=true.
   The upload UI should remain visible regardless of tier (upload is free).
   Upsell panel should appear in the sales section for users without the add-on,
   with a "Add Sales Tracking — $25/mo" button.
